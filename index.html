JFrog Platform Site Sizing Calculator

Sizing based on the JFrog self-managed reference architecture & hardware sizing matrix. Per-cloud instance types, replica counts, storage and co-location rules are pulled verbatim from JFrog's published tables.

Inputs

Selects the per-cloud instance catalog, managed-DB SKUs, storage classes and network guidance. Private DC mirrors the cloud CPU/RAM as generic VMs.
Kubernetes shows worker-node recommendations and a per-site cluster plan. Servers sizes one host per replica and generates an Ansible bundle.
Bare metal uses physical CPU cores (not vCPU) — sizing figures are identical, the label changes from vCPU to cores in the output.
Dedicated: one pod per node (mirrors VM model). Anti-affinity: pods can share a pool with other workloads, but podAntiAffinity keeps Artifactory replicas off the same node.
Single sizes one active cluster. Active + Passive adds a DR site (scaled by the passive option below). Active + Active adds a second full active site (both serve traffic, bidirectional federation). Both multi-site options report per-site footprints + a grand total.
Hot: identical replica count for instant failover. Warm: 1 replica per component (RabbitMQ 3 for quorum) with full DB; scale up on failover.
Total number of active HTTP threads serving requests across the entire Artifactory cluster at any given moment. Reference architecture tiers: Small ≤100, Medium ≤500, Large ≤1,200, XLarge ≤3,000, 2XLarge ≤6,000.
Total number of HTTP requests handled collectively by all nodes in the Artifactory cluster within a minute. Reference architecture tiers: Small ≤6K, Medium ≤50K, Large ≤100K, XLarge ≤200K, 2XLarge ≤500K.
Total artifact / binary footprint. Object storage is recommended in cloud.
A local SSD cache on each Artifactory node fronts the object store, serving hot binaries at local-disk latency. Recommended whenever binaries live in object storage.
Disabling cache-fs means every binary read goes directly to object storage. This significantly increases latency and egress cost for hot artifacts and is only suitable for on-prem or all-flash storage that is already local. Enable it unless you have a specific reason not to.
Local SSD cache per Artifactory replica, sized to hold the hot working set. Default 20% of the binary footprint.
Enables Xray, RabbitMQ, and JAS. When disabled, none of these are sized or deployed.
Drives Xray node size, RabbitMQ replica count and Xray DB sizing. Over 100K artifacts forces RabbitMQ split mode.
Scales the load drivers above (clients, RPM, storage, indexed artifacts) to size for projected growth. Set to 0 to size for today only.
Multi-replica uses JFrog-published replica counts per tier (1/2/3/4/6 for Artifactory).
Cloud-native (ALB/NLB, App Gateway, GCLB) or hardware/external LB (F5, HAProxy, etc.). When absent, NGINX handles TLS termination and routing directly.
JFrog NGINX handles TLS termination, routing, and large-upload buffering. Required when no external LB is present. Can be omitted when an LB routes directly to Artifactory's built-in router.
RabbitMQ is only deployed when Xray is enabled. Choosing External removes the RMQ nodes from this footprint — you provision and operate it separately; the recommended size + required plugins/config are shown in the results.
Distribution co-locates on Artifactory (+200 GB, no new VMs). Mission Control is bundled into Artifactory (no separate node or DB). JAS requires Xray; on VMs it adds dedicated servers, on Kubernetes it runs within the Xray Helm chart (no extra nodes). Workers add their own nodes per the hardware sizing matrix. AppTrust + UnifiedPolicy co-locate on Artifactory nodes (small overhead — no dedicated VMs or node pools). Curation adds the Catalog service (+ its own DB and a Valkey cache). Runtime Security adds a Runtime server (+ its own DB) and a per-node sensor DaemonSet — both as separate Helm releases.
Only applies when Curation is enabled. Co-located runs Valkey on the Catalog nodes (adds RAM, no new VMs); External uses a managed/standalone Valkey (or Redis) cluster — removed from the footprint, with a recommended size + config shown.
Co-located = a PostgreSQL provisioned with the deployment (the Helm chart's bundled DB on K8s, or DB nodes alongside the platform on VMs) — it counts toward the footprint. External = any managed/standalone RDBMS you run separately (RDS / Cloud SQL / Flexible Server / self-managed), sized here but outside the footprint.
Shared = one PostgreSQL instance hosting a logical database per product (the typical setup; the Helm chart's bundled DB works this way). Dedicated = a separate instance per product (more DB nodes). Each product always gets its own logical database either way.
