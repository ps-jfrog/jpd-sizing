JFrog Platform Deployment (JPD) Sizing Calculator

Sizing based on the JFrog self-managed reference architecture & hardware sizing matrix. Per-cloud instance types, replica counts, storage and co-location rules are pulled verbatim from JFrog's published tables.

Inputs

Selects the per-cloud instance catalog, managed-DB SKUs, storage classes and network guidance. Private DC mirrors the cloud CPU/RAM as generic VMs.
VMs size one host per replica. Kubernetes shows the same capacity as worker-node recommendations and adds a per-site cluster plan (node pools, sizing, control plane).
Dedicated: one pod per node (mirrors VM model). Anti-affinity: pods can share a pool with other workloads, but podAntiAffinity keeps Artifactory replicas off the same node.
Single sizes one active cluster. Active + Passive also sizes a DR site (scaled by the passive option below) and reports a grand total across both sites.
Hot: identical replica count for instant failover. Warm: 1 replica per component (RabbitMQ 3 for quorum) with full DB; scale up on failover.
Hardware sizing matrix tiers: ≤20, ≤100, ≤200, >200 (support).
Reference architecture tiers: Small ≤6K, Medium ≤50K, Large ≤100K, XLarge ≤200K, 2XLarge ≤500K.
Total artifact / binary footprint. Object storage is recommended in cloud.
A local SSD cache on each Artifactory node fronts the object store, serving hot binaries at local-disk latency. Recommended whenever binaries live in object storage.
Local SSD cache per Artifactory replica, sized to hold the hot working set. Default 20% of the binary footprint.
Drives Xray node size, RabbitMQ replica count and Xray DB sizing. Over 100K artifacts forces RabbitMQ split mode; "No Xray" omits Xray, RabbitMQ and JAS entirely.
Scales the load drivers above (clients, RPM, storage, indexed artifacts) to size for projected growth. Set to 0 to size for today only.
Multi-replica uses JFrog-published replica counts per tier (1/2/3/4/6 for Artifactory).
JFrog NGINX is the bundled reverse proxy (TLS, routing, large-upload buffering). Choose Load Balancer to front the platform with a cloud-native LB (ALB/NLB, App Gateway, GCLB) or an external/hardware LB (F5, HAProxy, etc.).
Route to Artifactory: the LB terminates TLS and hits Artifactory's built-in router — no NGINX VMs provisioned. Route to NGINX: keep the NGINX tier behind the LB for advanced rewrite/proxy features.
RabbitMQ is only deployed when Xray is enabled. Choosing External removes the RMQ nodes from this footprint — you provision and operate it separately; the recommended size + required plugins/config are shown in the results.
Distribution co-locates on Artifactory (+200 GB, no new VMs). JAS, Workers, AppTrust + UnifiedPolicy and Mission Control add their own nodes per the hardware sizing matrix. JAS requires Xray. Curation adds the Catalog service (+ its own DB and a Valkey cache).
Only applies when Curation is enabled. Co-located runs Valkey on the Catalog nodes (adds RAM, no new VMs); External uses a managed/standalone Valkey (or Redis) cluster — removed from the footprint, with a recommended size + config shown.
Managed uses the cloud's PostgreSQL service (sized here, provisioned outside the cluster). Self-hosted adds a primary you run (plus a standby when HA is on).
